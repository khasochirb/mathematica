- **HttpOnly refresh token cookie** (commit `eea7b31`, Option A scope). Refresh token moved from JS-readable cookie to server-set HttpOnly + Secure (prod) + SameSite=Lax + Path=/api/auth + Max-Age=2592000. New `/api/auth/logout` route added to clear the cookie (JS can't clear HttpOnly cookies). Login/register/refresh routes set the cookie via Set-Cookie; refresh route reads from cookie instead of body. Cookie attributes centralized in [lib/auth-cookies.ts](lib/auth-cookies.ts) so set-sites and clear-site can't drift. **Access token deliberately NOT migrated** (Option B was considered and rejected): access token expires in 1 hour and is rotated on every refresh via RTR, so the XSS blast radius is small (~1 hour); moving access token to HttpOnly would require rewriting `lib/use-performance.ts` direct-Supabase calls, add latency to every attempt write, and yield marginal security gain. Refresh token is the high-value asset and HttpOnly defends it.
